### 2026-07-13 - HTML email body (body_html) on send/draft/reply tools

- Completed: Fixed the second Cowork limitation. The send tools only produced Content-Type text/plain, so an HTML body (for example a styled table) arrived as literal tags. Added an optional body_html parameter to send_email, create_draft, update_draft, and reply_all (PR #23, `2750afb`). When body_html is provided, build_email_message does set_content(body_text) then add_alternative(body_html, subtype="html"), producing multipart/alternative (text/plain fallback first, text/html last). Omitting body_html is unchanged (single text/plain). body_text stays required as the plain fallback; body_html is optional and uncapped (symmetric with body_text; the 25 MiB assembled-message cap and the 50 MiB request middleware are the backstops). The outbound HTML is passed verbatim (not sanitized), since it is the recipient-rendered body, not a header. Works with attachments (multipart/mixed wrapping the multipart/alternative).
- Current state of the system: Merged to main (HEAD `2750afb`) and deployed to Railway (mcp-gmail service in the Welch-Products-Command-Center project); deploy success, live. 945 tests passing (+27), ruff clean, all SAST gates (Bandit/Semgrep/Trivy) green on the PR. Additive contract only (existing body_text-only callers are byte-identical; the test diff was 589 insertions, 0 deletions). No new dependency (standard-library email MIME). Tool count stays 33.
- Reviews: Crash Override approved and independently verified the MIME assembly plus that a hostile body_html carrying CRLF and fake headers cannot inject real message headers (structurally contained by Python's randomized MIME boundary and part-body encoding; header validation stays headers-only, which is why the body is intentionally not sanitized). Codex cross-model gate (send path, third-party-API submission, contract change) returned No issues found.
- Next: none required. Cowork can now send styled HTML inline via body_html. Note: the client must reconnect the Gmail MCP integration to see the new body_html param and the earlier format="text" enum in the advertised tool schema (MCP clients cache tools/list at connect; the server already accepts the new inputs).
- Open decisions: none. Minor optional polish both reviewers noted: an explicit body_html="" produces an empty html part (the plain fallback stays intact, so benign); a future tweak could treat empty-string as plain-only.
- Estimate vs actual: moderate additive send-path feature. Agent compute ran a little longer than a plain feature build because of two 300-LOC-rule deviations handled mid-build (tool_router_write.py accepted at 302 as a single-responsibility dispatch table; tool_definitions_write.py trimmed back under 300 after an over-verbose schema description). Roughly 45 to 70 minutes agent compute across the build, Crash Override review, the Codex gate, and the merge.

### 2026-07-12 - read_email + get_thread format="text": token-efficient plain-text read (Cowork bloat fix)

- Completed: Fixed the friction a Claude Cowork session hit. read_email (format="full") returns the entire raw message including the full HTML part and inline base64, so bloated Amazon receipt emails (170K to 250K chars) blow past the MCP output token cap and get dumped to a file. Added a new format="text" value to read_email AND get_thread (PR #21, `08d9d1d`): the server fetches format=full from Gmail, then returns a lean object (id, threadId, labelIds, snippet, curated headers, text, text_source, attachments as metadata only with no bytes) containing just the decoded plain-text body. Prefers the text/plain part (charset-aware decode); if none, converts text/html via markdownify (new MIT dependency); if neither, text_source="none". Robustness: hostile deeply-nested HTML degrades to empty text instead of raising RecursionError (honoring the module's never-crashes-the-read contract); get_thread fault-isolates each message so one bad message cannot fail the whole thread; a defensive text cap (MAX_TEXT_CHARS = 100000) with a truncation marker and a text_truncated flag bounds output. New modules message_text.py + message_text_parts.py; tool_definitions.py was split (3 thread-management defs moved to a new tool_definitions_threads_manage.py) to stay under the 300 LOC rule.
- Current state of the system: Merged to main (HEAD `08d9d1d`) and deployed to Railway (mcp-gmail service in the Welch-Products-Command-Center project); deploy success, live. 918 tests passing, ruff clean, all SAST gates (Bandit/Semgrep/Trivy) green on the PR (dogfooding the CI scanning added earlier this session). No migration, no blast-radius change (additive read-path feature; existing formats full/metadata/minimal/raw unchanged). Tool count stays 33 (a format value was added, no new tool).
- Reviews: Crash Override reviewed the feature, caught a real robustness gap (hostile HTML raised RecursionError and one bad message failed a whole get_thread), and re-verified the fix. With that untrusted-input edge closed and tested it dropped to the routine tier, so no Codex cross-model gate. A licensing catch during grounding: the originally-planned html2text is GPL-3.0-or-later, incompatible with this MIT-distributed repo, so it was swapped for markdownify (MIT). New dependency: markdownify (MIT, free, no API key, no network); transitives beautifulsoup4/soupsieve/six, all hash-pinned in both lockfiles.
- Next: none required. Optional tunes: lower MAX_TEXT_CHARS if the worst-case single-message output (about 25K to 33K tokens at the 100K-char cap) needs more headroom under the client token cap; extend format="text" to other body-returning read tools (batch_read_emails, multi_search_emails) if the bloat surfaces there.
- Open decisions: MAX_TEXT_CHARS = 100000 default. Crash Override noted it is a defensive ceiling against multi-MB bodies, not a guaranteed sub-token-cap for a single maximally-truncated message; it is a one-line tune.
- Estimate vs actual: moderate feature, agent compute ran longer than a plain feature build because of two clean module splits (the 300 LOC rule), the html2text to markdownify license correction caught in grounding, and one Crash-Override-found robustness round (hostile-HTML degrade + per-message thread isolation + text cap). Roughly 60 to 90 minutes agent compute across the build, three Crash Override passes, and the merge.
- SURFACE TO ADVISORY: license-compatibility check worth adopting project-wide. A planned dependency (html2text) turned out to be GPL-3.0-or-later, which conflicts with this repo's MIT license since the server is distributed. Recommend that dependency additions across Welch projects check the new package's license against the project's own license, and prefer permissive (MIT/BSD/Apache/PSF) for distributed/MIT projects. markdownify (MIT) was the clean substitute here.

### 2026-07-06 - Upload-slot + handle flow for large email attachments (barcode-safe)
- Completed: Added an out-of-band upload path so large email attachments never pass through the model as base64 (which corrupted barcodes on shipping-label PDFs). New MCP tool create_attachment_upload_slot mints a one-time capability token (secrets.token_urlsafe(32), SHA-256 at rest, 15-min TTL, bound to (auth0_sub, account_email), 10-slot per-user cap). New public endpoint POST /attachments/upload receives raw bytes with the token in an X-Upload-Token header (validated before the body is read), enforces a hard 25 MiB streaming cap and a 100 MiB per-user cap, and stores the bytes Fernet-encrypted in the new attachment_uploads table (migration 0004, NULL-on-consume, startup+hourly purge). send_email/create_draft/update_draft/reply_all accept a oneOf-tagged attachment union: inline {filename,mime_type,data_base64url} or {source:"upload",upload_token}. Slots are consumed only AFTER a successful build_email_message (exact assembled-size gate <=25 MiB), before the Gmail POST, atomically all-or-nothing; the idempotency cache read precedes consume. Input hardening: control chars rejected in filename and mime_type at every ingress; a duplicate upload_token in one message rejected before consume; every handler maps a build ValueError to a typed bad_request (no 500). Two module splits for the 300-LOC rule: attachment_input.py, drafts_lifecycle.py (re-exported so no import paths changed). Tool count 32 to 33.
